1. SUMÁRIO EXECUTIVO

O InspFLOW é uma aplicação de orquestração de processos de manutenção, inspeção e passagem de serviço, desenvolvida sobre a Microsoft Power Platform. A solução atua como uma interface centralizadora (Front-end) sobre um “banco de dados” descentralizado no SharePoint Online (SPO).

Diagnóstico Atual: O sistema é operacional e resolve dores imediatas de comunicação. Contudo, opera sob uma arquitetura de dados fragmentada ("Sharding Manual"), onde cada Unidade/Parada exige a criação física de novas listas e planilhas de ZR’s, gerando dívida técnica acumulada, esforço manual de manutenção e riscos de integridade referencial.

2. ARQUITETURA DE DADOS (BACK-END)

A arquitetura não segue o modelo relacional tradicional (Tabela Única de Fatos).
O sistema utiliza o padrão de "Cluster de Listas por Evento".

2.1 O Conceito de Cluster (Unidade de Persistência)

Para cada Parada de Manutenção ativa, o sistema exige a existência de um conjunto (Cluster) de 4 listas SharePoint independentes. O aplicativo decide em qual conjunto gravar/ler baseado em variáveis de contexto selecionadas pelo usuário no login.
Hoje essa criação está sendo realizada pelo Power Automate (Que ainda não tive acesso aos fluxos)

3. ESPECIFICAÇÃO DO FRONT-END (PowerApps)

3.1 Mecanismo de Navegação

  • Interface: Tablet Layout (Otimizado para Desktop).
  • Context Switching: Ao iniciar (OnStart), o App verifica a preferência do usuário ou força a seleção de unidade. Baseado na escolha, define variáveis globais que "apontam" para o Cluster de listas correto.
  • Menu Lateral: Persistente, com rotas para: Passagem de Serviços, Inspeção, Manutenção, Equipamentos, Equipe, Visão Admin, Visão Planejador e Painel PBI.

3.2 Dashboard Operacional (Swimlanes)

A tela inicial ("Home") utiliza lógica de filtragem cruzada em três vetores visuais:

  1. Notificações: Query na lista _Eventos (Filtro: Menções ao usuário logado).
  2. Manutenção: Query na lista _Acompanhamento (Filtro: Responsável = Usuário & Status != Concluído).
  3. Inspeções: Query na lista _Acompanhamento (Filtro: Tipo = Inspeção & Status = Pendente).

3.3 Delegação e Performance

Devido ao uso de listas grandes (ex: tb_Equipamentos_GISNPEQ com 6.500+ itens), o sistema opera acima do limite de delegação padrão.

  • Impacto: Filtros de texto do tipo "Contém" (in) podem não retornar todos os resultados se os dados não estiverem indexados ou cacheados em Coleções locais.

4. INTEGRAÇÕES EXTERNAS

  • GINSPEQ: Deep links ("Links Profundos") permitem navegação direta para a ficha técnica do equipamento no sistema corporativo via TAG.
  • Power BI: O módulo "Painel Gerencial PBI" exibe relatórios embarcados, consumindo dados das mesmas listas SharePoint (provavelmente via Gateway ou atualização agendada).

5. MATRIZ DE RISCO E DÍVIDA TÉCNICA (AS-IS)

Nível

Componente

Descrição do Risco

Impacto no Negócio

CRÍTICO

Escalabilidade (Sharding)

A criação de uma nova parada exige criar 4 novas listas manualmente no SharePoint.

Dependência total do Desenvolvedor. Não é "Self-Service". Risco de erro humano na criação.

ALTO

Integridade Referencial

Vínculos entre tabelas são feitos por Texto (TAG), não ID único.
(Subterfúgio para não permissionamento - PTB)

Se a Engenharia renomear um TAG, o histórico de inspeção e manutenção daquele equipamento se perde.

ALTO

Dependência de Excel

Uso de planilhas (ListasZR.xlsx) como banco de dados.

Arquivos podem ser bloqueados, deletados ou ter colunas renomeadas, causando "Crash" imediato no App.

MÉDIO

Limpeza de Código

Presença de artefatos de outras unidades (REFAP) e listas de teste (LabEventos).

Confusão na manutenção e "peso" desnecessário no carregamento do App.

Foco: Eficiência operacional.

  1. Funcional:

    (1. Dados > 2. Análise > 3. Conclusão).

    (Talvez impacto de desempenho forte)
  1. Higiene de Dados:
  • Eliminar Excel: Migrar dados estáticos (ZR’s) para Listas SharePoint
    (Validar possibilidade em questão de formatação fora do SAP)
  • Auditoria de Permissões: Restringir acesso ao Site SharePoint
    (Evitar deleção acidental de listas)

FASE 3: ESTRUTURAÇÃO E ARQUITETURA

Foco: Escalabilidade e redução de manutenção.

  1. A Grande Migração (Unificação):
  • Criar listas mestras globais (TB_Global_Acompanhamento) com coluna ID_Parada.
  • Mitigar a necessidade de criar listas novas para cada evento. (Se possível)
  • Benefício: Power BI lê uma fonte única; Manutenção zero para novas paradas.
  1. Automação Avançada:
  • Substituir e-mails por Adaptive Cards no Teams para aprovações.
  • Geração automática de PDF de Handover de Turno.

7. ANEXO: PROCEDIMENTOS CRÍTICOS VISUALIZADOS

7.1 Como Adicionar Nova Parada (Procedimento Atual)

  1. No SharePoint: Criar as 4 listas copiando a estrutura exata de um cluster existente (usar PowerAutomate ainda funcional via criação de parada no app).
  2. Nomear rigorosamente no padrão: [NOME]_[MODULO].
  3. Cadastrar o [NOME] na lista DadosParada.
  4. Validar se o App reconhece o novo contexto.
  5. Validar o tipo de colunas das listas criadas. (comum dar erro)

7.2 Contingência (Em caso de falha)

Se o App parar de carregar dados de equipamentos:

  1. Verificar se o arquivo GINSPEQ_INSPFLOW.xlsx na biblioteca "Documentos" está aberto por alguém (Check-out).
  2. Solicitar fechamento imediato do arquivo.
